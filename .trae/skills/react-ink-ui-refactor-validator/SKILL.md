# React/Ink UI Refactor Validator

## 适用场景

当你遇到以下任务时调用本 skill：

- React 或 Ink 组件被编译产物、自动生成代码、混乱状态逻辑污染，需要还原成可维护代码
- 需要确认“重构后是否与原行为等价”
- 组件存在 feature gate、配置开关、运行时守卫，人工判断容易漏条件
- 需要把状态逻辑抽成纯函数，以便补单元测试
- 需要补一层更贴近实际渲染的 UI 测试，而不是只测纯函数
- 用户说“类似代码还有很多需要更新”，希望形成稳定可复用流程

## 目标

对目标组件执行“行为不变优先”的重构，并建立双层验证：

1. 状态层：纯函数或轻量逻辑测试
2. 渲染层：真实 React/Ink 渲染输出测试

## 工作原则

- 先保行为，再提可读性
- 先找运行时入口，再改组件内部
- 优先抽离可复用、可单测的状态逻辑
- 对 feature gate、配置、全局状态做显式检查
- 组件测试尽量用真实渲染器，不只依赖浅层 mock
- 变更尽量小而清晰，避免顺手改无关逻辑

## 推荐流程

### 1. 建立行为基线

- 查看提交差异、调用入口、被谁渲染
- 标出组件的输入来源：
  - props
  - 全局状态
  - feature flag
  - 配置
  - 定时器
  - effect
- 列出必须保持不变的外部行为：
  - 显示条件
  - 消失条件
  - 定时阈值
  - 文本内容
  - 动画/样式变化
  - 切换场景下的重置逻辑

### 2. 识别可抽离状态逻辑

优先寻找以下模式并抽成纯函数：

- `reaction` / `status` / `mode` 切换时需要重置局部状态
- `tick` / `step` / `age` / `phase` 推进逻辑
- fading、过期、显示窗口等阈值判断
- 派生状态计算

抽离后应至少提供：

- 初始状态创建函数
- 同步状态函数
- 推进状态函数
- 派生判断函数

### 3. 为 UI 层建立稳定测试点

若原组件依赖过多运行时上下文，不要直接硬测整个容器组件。优先：

- 把最终展示层提取成一个小的展示组件
- 展示组件只接收：
  - 文本
  - 颜色/rarity
  - tick 或派生布尔值
  - tail / layout 之类纯展示参数
- 容器组件保留：
  - feature gate
  - store 读取
  - config 检查
  - timer/effect

这样可以把“逻辑正确”和“UI 确实渲染”分层验证。

### 4. 测试设计模板

#### 状态层测试

至少覆盖：

- 初始状态正确
- 输入不变时不误重置
- 输入变化时正确归零/切换
- 定时推进正确
- 阈值边界正确
- 从 A 切到 B 时不会继承 A 的过期/淡出状态

#### UI 层测试

至少覆盖：

- 文本是否真的渲染出来
- 关键布局元素是否存在
- 关键样式在边界前后是否发生变化
- 必要时使用真实渲染器输出缓冲区，而不是只看字符串快照

### 5. 验证顺序

始终按以下顺序验证：

1. 新增目标测试
2. 跑目标测试文件
3. 跑项目要求的 lint
4. 若项目有 typecheck，再跑 typecheck

如果环境缺少命令，要明确记录阻塞点与已完成的静态验证。

## React/Ink 场景专项建议

### A. 处理 feature-gated 组件

- 不要假设组件渲染失败就是逻辑错
- 先检查：
  - `feature('FLAG')`
  - 配置项是否允许显示
  - 全局状态是否提供了必要输入
  - 调用入口是否只在特定布局下挂载

### B. 处理定时器

- 对“时间驱动 + UI 渲染”场景，优先把 tick 逻辑抽到纯函数
- UI 测试只验证：
  - 文本
  - 样式变化
  - 结构元素
- 不要把所有计时行为都强压到组件测试里

### C. 处理 Ink 组件

- 优先使用真实 Ink 渲染工具，而不是只 mock 文本组件
- 若 stdout 抓帧不稳定，可转为 screen buffer 级别断言
- 如果容器组件受运行时守卫影响太大，抽取展示组件单独测

## 输出要求

完成任务后，应向用户明确说明：

- 哪些行为被视为“必须保持等价”
- 抽离了哪些纯函数
- 新增了哪些测试层次
- 实际执行了哪些验证命令
- 哪些结论是“静态判断”，哪些结论是“测试已证明”

## 可直接套用的任务清单

1. 找调用入口和显示守卫
2. 列行为不变量
3. 抽纯函数
4. 缩小展示组件边界
5. 补状态测试
6. 补 UI 渲染测试
7. 跑目标测试
8. 跑 lint/typecheck
9. 汇总结论与剩余风险

## 反模式

避免以下做法：

- 只看 diff 就宣布“逻辑等价”
- 不抽逻辑，直接在复杂容器组件上堆大量 mock
- 只写快照测试，不测边界条件
- 把 feature gate 绕过后就当成真实验证
- 修改太多无关代码，增加审查成本

- “确认重构前后行为一致，并写验证代码”
